Mach 设计原则

10.1.1 Mach 设计原则

  1. Mach 是什么？
    简单、最小的核心，支持 面向对象模型，独立的组件（子系统）之间可以通过 消息的方式互相通信。
    mach 本身提供模型，操作系统依赖模型(特性)实现。

xnu 是 unix(BSD 4.4)在 mach 的具体实现.

重点关注 Mach 模型（特性）的应用

  • 面向对象的特点
  • 消息通信的本质
  1. Mach中所有东西都是通过自己的对象实现。比如：进程、线程、虚拟内存都是对象，对象都有自己的属性。
  2. Mach 的独特之处在 通过消息传递的方式实现对象间的通信。Mach 不能直接调用另一个对象，而必须传递消息。src 对象发送一条消息，消息被加入 dst 对象的消息队列中等待处理。类似地，消息处理中产生一个应答，此应答通过另一条消息发送回src对象。

消息以 FIFO 的方式存储。
消息的内容 完全有发送者和接受者自己协商。

xun 目录- osfmk/man 目录 和 CMU 的原始文档

10.1.2 Mach 设计目标

Mach 只提供底层的仲裁原语(Mach 会提供施行策略的方法，但不会提供策略本身)
打个比方：现在有一个妹子很漂亮想和她去看电影，那么就有多种策略来约到她，比如：诗情画意打动她，或者英雄救美。那么实施这些策略的方法是：1.接近妹纸 2.互相说上话 3.约时间看电影。
这些方法还可以再具体到时间，地点，场景等等。那么 Mach就是这里实现策略的方法。

10.2 Mach消息

Mach 消息的设计考虑了 参数串行化，对齐，填充和字节顺序的问题，这些问题都被消息实现影藏了。

消息尾

消息还可以 选择 带一个消息尾（trailer），trailer 的类型是 mach_msg_trailer_type_t （unsigned int）表示。

每种type 对应一种特殊的 trailer 格式，这些格式可以为以后实现扩展使用。

trailer type usage
mach_msg_trailer_t 空 trailer
mach_msg_security_trailer_t 发送者安全令牌
mach_msg_seqno_trailer_t 顺序编号
mach_msg_xxxx_trailer_t

复杂消息

复杂消息和 简单消息的主要区别在 消息描述符.
简单消息使用mach_msg_type_descriptor_t 结构体定义 消息描述符；
复杂消息使用mach_msg_ool_descriptor_t结构体定义 消息描述符。

特殊的Port

每个 Task 中都会从 launchd 中继承 5种特殊的 Port.

  • kernel port
  • debug control port
  • host port : 代表正在运行该 task 的机器的 port
  • bootstrap port : 和bootstrap server连接着的一个port
  • task port : task 本身的 port
    重点关注host/task/bootstrap port。
